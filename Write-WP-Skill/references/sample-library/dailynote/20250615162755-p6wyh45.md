# 145 `Why it can UAF?`

glibc在选择空闲的堆块时，如果存在一个空闲的堆块，且其足够大，则malloc会选择它，并申请出来

我们首先申请两个堆块

```C
a = malloc(0x512); // 0x86a260
b = malloc(0x256); // 0x86a780
```

将 "AAAAAAAA" 写入 a 中

此时 **0x86a260** 的堆块，指向 **"AAAAAAAA"**

这时，我们将  **a**堆块  free掉

接下来，我们随意申请一个小于 0x512 的堆块，就会分配到原来的 a 处

```C
c = malloc(0x500);
```

此时得到的 c堆块 的地址是 **0x86a260**

这时，我们向 c堆块 中写入 "CCCCCCCC"

则此时输出 c堆块以及a堆块，输出的都会是 "CCCCCCCC"

这就是初步的UAF漏洞

# 146 `Why it can UAF?`

我们首先申请到一个 0x20大小的内存 p1,将 p1[1] 赋值为 printf 函数，打印"Hello CTF_show"

```C
typedef void (*ptr)(char *);
p1 = (ptr *)malloc(0x20); // 0x1aee010
p1[1] = (ptr)printf;
p1[1]("Hello CTF_show");
```

此时程序运行一切正常，输出的也是 "Hello CTF_show"

此时将 p1 free 掉，由于此时 p1 未置空，p1[1]中仍存储的为 printf 函数，此时仍然可以进行输出的操作

```C
p1[1]("Print Again");
```

这一段的回显为 "Print Again"

这时，我们再进行malloc操作，将会将刚刚free掉的p1分配出来

```C
p2 =  (ptr *)malloc(0x20); // 0x1aee010
```

此时， p1_addr == p2_addr

此时我们如果修改p2[1]为demoflag,即system函数，再去执行p1("/bin/sh")，便会得到shell

```C
p2[1] = (ptr)demoflag
p1[1]("/bin/sh") #system("/bin/sh")
```
